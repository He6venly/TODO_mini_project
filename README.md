# TODO_mini_project
소프트웨어공학 미니 프로젝트 TODO ListApp

### 미니 프로젝트 보고서: MongoDB 연결 트러블슈팅

#### 1. 어려웠던 점 (Troubleshooting)
프로젝트 초기 환경 세팅 중, MongoDB Atlas와 백엔드 서버(Node.js/Express)를 연결하는 과정에서 두 가지 큰 에러에 직면하며 상당한 시간을 소요했습니다.

* **이슈 1: 특수기호로 인한 URI 파싱 에러 (`got "undefined"`)**
  * **원인:** DB 비밀번호에 `@` 기호가 포함되어 있었는데, MongoDB 연결 주소(`mongodb+srv://아이디:비밀번호@주소`) 체계에서 `@`는 아이디/비밀번호와 서버 주소를 구분하는 특수 예약어로 사용됩니다. 이로 인해 컴퓨터가 주소의 끝을 제대로 인식하지 못해 에러가 발생했습니다.
  * **해결:** 비밀번호 안의 `@` 기호를 URL 인코딩 방식인 `%40`으로 치환하여 파싱 에러를 해결했습니다.

* **이슈 2: 네트워크 DNS 차단으로 인한 연결 거부 (`querySrv ECONNREFUSED`)**
  * **원인:** 비밀번호 문제를 해결했음에도 서버가 데이터베이스를 찾지 못하고 튕겨내는 현상이 발생했습니다. 확인 결과, 최신 MongoDB 연결 방식인 SRV 레코드(`mongodb+srv://`)를 현재 사용 중인 네트워크(또는 통신사/방화벽) 환경에서 DNS 조회 단계를 차단하고 있었기 때문입니다.
  * **해결:** MongoDB Atlas 대시보드에서 최신 Node.js 드라이버 대신 구버전(2.2.12 or later)을 선택하여, SRV를 거치지 않고 각 샤드(Shard) 서버에 직접 연결하는 **레거시 연결 문자열(`mongodb://`)**을 발급받아 우회 접속에 성공했습니다.

* **이슈 3: 프론트엔드 Tailwind CSS 초기화 명령어 에러 (`could not determine executable to run`)**
    * **원인:** React 프론트엔드 환경 세팅 중 `npx tailwindcss init -p` 명령어를 실행했을 때 실행 파일을 찾을 수 없다는 에러가 발생했습니다. 확인 결과, 최근 Tailwind CSS가 v4로 메이저 업데이트되면서 기존 v3에서 사용하던 `init` CLI 명령어 방식이 변경/삭제되어 발생한 버전 호환성 문제였습니다.
    * **해결:** 설치된 최신 v4 패키지를 삭제(`npm uninstall`)하고, 명령어에 버전 태그를 명시하여 안정적인 기존 버전(`npm install -D tailwindcss@3`)으로 다운그레이드 재설치한 후 명령어를 실행하여 해결했습니다.

* **이슈 4: Vercel 배포 빌드 에러 및 OS 간 대소문자 인식 차이 (`No such file or directory`)**
  * **원인:** 로컬 개발 환경(Windows)에서는 폴더명을 대문자가 포함된 `frontEnd`, `backEnd`로 생성했음에도 소문자로 경로를 호출할 때 문제가 없었습니다. 그러나 배포 환경인 Vercel 서버(Linux)는 파일 시스템이 대소문자를 엄격하게 구분하므로, 소문자로 작성된 빌드 명령어가 실제 폴더를 찾지 못해 빌드 실패와 404 에러를 유발했습니다.
  * **해결:** `vercel.json` 설정 파일 및 빌드 스크립트 내부의 모든 경로를 실제 폴더명과 완벽하게 일치하는 대소문자(`frontEnd`, `backEnd`)로 수정하여 경로 탐색 오류를 해결했습니다.

* **이슈 5: 프론트엔드 정적 파일 빌드(`dist`) 및 라우팅 매핑 오류 (지속적인 404 Not Found)**
  * **원인:** 빌드 성공 후에도 웹사이트 접속 시 404 에러가 발생했습니다. Vite를 통해 프론트엔드를 빌드하면 결과물이 `dist` 폴더에 생성되는데, Vercel의 `@vercel/static-build` 모듈이 이 폴더의 구조를 처리하는 방식과 `vercel.json`에 명시된 라우팅(`rewrites`) 경로가 엇갈린 것이 원인이었습니다.
  * **해결:** Vercel이 내부적으로 정적 파일을 제공하는 경로 규칙에 맞게, `vercel.json`의 목적지(destination) 경로에서 `/dist/` 부분을 제거하고 `/frontEnd/$1`로 경로를 직접 매핑하여 화면이 정상적으로 렌더링되도록 조치했습니다.

* **이슈 6: 프론트엔드에서 추가한 데이터(D-Day, 수정 시간)가 DB에 저장되지 않고 증발하는 현상**
  * **원인:** 프론트엔드 화면에 D-Day 입력 달력을 만들고, MongoDB의 설계도(Schema)에 `dueDate`와 `editedAt` 공간을 새로 추가하는 것까지는 성공했습니다. 그러나 정작 프론트엔드와 DB를 연결해 주는 **백엔드 API 라우터(`app.post`, `app.put`)** 코드는 과거의 형태에 머물러 있었습니다. 이로 인해 백엔드가 새로 넘어온 데이터들을 인식하지 못하고 무시(누락)한 채 제목(`title`)만 DB에 저장하고 있었습니다.
  * **해결:** 데이터가 화면에서 DB까지 흘러가는 전체 파이프라인(Full-Stack)의 구조를 이해하고, 백엔드 API 라우터가 `dueDate`와 `editedAt` 속성을 명시적으로 받아 MongoDB에 전달하도록 코드를 업데이트하여 데이터 누락 문제를 해결했습니다. 
  * **핵심 인사이트 (프롬프트 엔지니어링의 한계와 도메인 지식의 필요성):** 이 과정에서 **"AI에게 단순히 'D-Day 기능을 만들어줘'라고 프롬프트를 입력하는 것만으로는 완벽한 결과물을 얻을 수 없다"**는 것을 깨달았습니다. 프론트엔드(화면), API(전달자), DB(저장소)로 이어지는 웹의 전반적인 아키텍처와 '데이터베이스 스키마 마이그레이션' 같은 도메인 지식이 있어야만, AI가 생성한 코드 중 어느 부분이 누락되었는지 파악하고 정확한 추가 수정을 지시할 수 있음을 체감했습니다.

* **이슈 7: 할 일 항목 클릭 시 원치 않는 '완료' 처리 발생 (이벤트 전파 문제)**
  * **원인:** 할 일 항목의 빈 공간이나 글자를 드래그하려 할 때마다 체크박스를 누른 것과 동일하게 '완료' 처리가 되는 불편함이 있었습니다. 확인 결과, 클릭 이벤트 핸들러(`onClick`)가 체크박스 자체가 아닌, 전체를 감싸는 부모 컨테이너(`div`)에 부여되어 있었기 때문입니다.
  * **해결:** 부모 태그의 `onClick` 이벤트를 제거하고, 실제 체크박스(`input type="checkbox"`) 태그에만 `onChange` 이벤트를 부여하여 클릭 가능한 영역(Clickable Area)을 엄격하게 제한함으로써 사용자 경험(UX)을 개선했습니다.


#### 2. 생성형 AI 활용 방법 (AI 어떻게 썼나)
* **에러 로그 분석 및 원인 고립화:** 터미널에 출력된 복잡한 콜스택 에러 로그(`querySrv ECONNREFUSED _mongodb._tcp...`)를 통째로 AI에게 입력하여, 문제의 원인이 단순 오타가 아닌 '네트워크 DNS 차단'에 있다는 것을 빠르고 정확하게 진단받았습니다.
* **단계별 테스트 주도:** `.env` 파일의 문제인지, DB 설정 문제인지 파악하기 위해 AI의 가이드에 따라 하드코딩 테스트, 핫스팟 연결, IP `0.0.0.0/0` 허용 확인 등을 단계적으로 수행하며 변수를 통제하는 디버깅 방법을 배웠습니다.

* **우회 해결책 탐색:** 네트워크 방화벽 문제라는 데드락에 빠졌을 때, AI가 제시한 '레거시 연결 주소(Legacy Connection String) 사용'이라는 구체적인 우회 솔루션을 통해 문제를 돌파할 수 있었습니다.

* **최신 기술 생태계 이슈 파악:** 명령어를 정확히 입력했음에도 알 수 없는 에러가 발생했을 때, AI를 통해 이것이 내 컴퓨터의 문제가 아니라 '최근 발생한 라이브러리의 메이저 버전 업데이트' 때문이라는 최신 동향을 빠르게 파악할 수 있었습니다. 이를 통해 삽질하는 시간을 획기적으로 줄였습니다.

* **운영체제(OS) 수준의 맹점 파악:** Vercel 빌드 로그(`sh: line 1: cd: frontend: No such file or directory`)를 AI에게 분석하게 하여, 단순한 오타가 아니라 'Windows와 Linux의 파일 시스템 대소문자 처리 방식 차이'라는 주니어 개발자가 놓치기 쉬운 OS 레벨의 원인을 빠르고 정확하게 짚어낼 수 있었습니다.

* **CI/CD 인프라 및 라우팅 구조 이해:** 배포된 결과물이 도저히 보이지 않는 데드락 상태에서, AI를 통해 Vite의 `dist` 빌드 결과물과 Vercel의 정적 라우팅 동작 원리(포장 껍데기와 알맹이의 비유)를 이해하고 올바른 `vercel.json` 설정값을 도출했습니다.

* **풀스택 데이터 플로우 설계:** 단순히 예쁜 화면을 만드는 것을 넘어, 프론트엔드의 `State`가 백엔드 API를 거쳐 MongoDB `Schema`에 안전하게 안착하기까지의 전체 데이터 흐름을 AI와 함께 설계하고 코드로 구현했습니다.

#### 3. 배운 점 (Lessons Learned)
* **환경 변수와 인코딩의 중요성:** 비밀번호 같은 민감한 데이터에 특수문자가 들어갈 경우, 시스템이 이를 예약어로 오인하지 않도록 반드시 URL 인코딩(Percent-encoding) 처리를 해야 한다는 기초적이지만 중요한 사실을 체감했습니다.

* **에러를 대하는 시야 확장:** 에러의 원인이 내 코드(`index.js`나 `.env`)에만 있는 것이 아니라, 사용하는 네트워크 환경, 방화벽, 심지어 통신사의 DNS 정책 설정 등 인프라적인 외부 요인에도 있을 수 있다는 시스템적인 시야를 갖게 되었습니다.

* **소프트웨어 공학적 문제 해결:** 문제가 발생했을 때 당황하지 않고, 현상을 분석하여 하나씩 변수를 소거해 나가는 체계적인 디버깅 프로세스를 몸소 경험했습니다.

* **버전 관리의 중요성과 생태계의 변화 속도:** 프론트엔드(NPM) 생태계는 변화 속도가 매우 빨라서, 불과 몇 달 전의 가이드라인이나 튜토리얼 명령어라도 라이브러리 업데이트에 따라 갑자기 작동하지 않을 수 있다는 점을 뼈저리게 느꼈습니다. 패키지를 설치할 때 무작정 최신 버전을 쓰기보다는 검증된 버전을 명시하여(예: `@3`) 설치하는 등 패키지 의존성 관리의 중요성을 배웠습니다.

* **로컬 환경과 프로덕션 환경의 괴리:** 내 컴퓨터(Local)에서 완벽하게 작동하는 코드가 서버(Production)에서는 OS나 파일 시스템의 차이로 인해 완전히 다른 결과를 낳을 수 있다는 점을 뼈저리게 경험했습니다. 

* **설정 파일의 중요성과 배포 파이프라인 이해:** 코드를 작성하는 것만큼이나, 작성된 코드를 클라우드 서버에 올리고 길을 터주는 설정 파일(`vercel.json`, `package.json`)의 역할이 CI/CD 파이프라인에서 얼마나 중요한지 구조적으로 이해하게 되었습니다.

* **AI를 100% 활용하기 위한 개발자의 도메인 지식:** AI는 강력한 코딩 도우미지만, '무엇을 질문해야 하는지' 모르면 근본적인 문제를 해결할 수 없습니다. 프론트엔드와 백엔드가 어떻게 통신하는지에 대한 아키텍처 지식이 있어야 AI가 뱉어낸 파편화된 코드를 하나의 완성된 서비스로 조립할 수 있음을 배웠습니다.

* **사용자 경험(UX)을 결정짓는 디테일:** 단순히 '작동하는 기능'을 넘어, D-Day 임박 시 강조되는 배지, 직관적인 '🎉 완료!' 애니메이션, 그리고 오작동을 막는 이벤트 영역 제한(`onChange`) 등 사소해 보이는 프론트엔드의 디테일이 실제 서비스의 퀄리티를 어떻게 끌어올리는지 체감했습니다.

* **DB 스키마 마이그레이션 경험:** 서비스 운영 도중 새로운 기능(D-Day, 수정 시간)이 추가되었을 때, 기존 데이터를 유지하면서 새로운 데이터 구조를 데이터베이스에 안전하게 추가하고 API를 연결하는 실무적인 확장 과정을 직접 경험했습니다.
