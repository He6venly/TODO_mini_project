# 📝 TODO_mini_project
**소프트웨어공학 미니 프로젝트: Full-Stack TODO List App**

🔗 **Live Demo (배포 링크):** [https://todo-mini-project-three.vercel.app](https://todo-mini-project-three.vercel.app)

<br>

## ✨ 프로젝트 개요 및 주요 기능 (Features)
React(프론트엔드), Node.js/Express(백엔드), MongoDB(데이터베이스)를 활용하여 밑바닥부터 구축한 풀스택(Full-Stack) 웹 애플리케이션입니다. Vercel을 통해 CI/CD 파이프라인을 구축하여 자동 배포되도록 설계되었습니다.

* 📅 **D-Day 및 실시간 시계:** 목표 날짜를 설정하면 오늘을 기준으로 남은 일수를 자동 계산하여 직관적인 배지(D-Day, D-N)로 시각화합니다.
* 🕒 **등록 및 수정 시간 추적:** 항목이 처음 생성된 시간뿐만 아니라, 내용이 수정되었을 때만 '수정 시간'을 별도로 기록하여 히스토리를 관리합니다.
* 📱 **반응형 웹 디자인 (Responsive):** 모바일, 태블릿, 데스크톱 등 접속하는 디바이스의 화면 크기에 맞춰 레이아웃과 UI 요소들이 유연하게 변환됩니다.
* 🎉 **사용자 친화적 UX/UI:** 창원대 시그니처 컬러를 테마로 적용하였으며, 할 일 완료 시 통통 튀는 애니메이션 피드백, 텍스트 오작동 방지(체크박스 전용 클릭) 등 디테일한 사용성을 고려했습니다.

### 🛠️ 1. 개발 단계별 주요 트러블슈팅 (Troubleshooting)

#### Phase 1. 데이터베이스(MongoDB) 및 백엔드 파이프라인 구축

* **특수기호 파싱 에러 (`got "undefined"`)**
  * **원인:** DB 비밀번호에 `@` 기호가 포함되어 있었는데, MongoDB 연결 주소(`mongodb+srv://아이디:비밀번호@주소`) 체계에서 `@`는 아이디/비밀번호와 서버 주소를 구분하는 특수 예약어로 사용됩니다. 이로 인해 컴퓨터가 주소의 끝을 제대로 인식하지 못해 연결이 튕기는 에러가 발생했습니다.
  * **해결:** 비밀번호 안의 `@` 기호를 URL 인코딩(Percent-encoding) 방식인 `%40`으로 치환하여 파싱 에러를 깔끔하게 해결했습니다.

* **네트워크 DNS 차단 우회 (`querySrv ECONNREFUSED`)**
  * **원인:** 비밀번호 문제를 해결했음에도 서버가 DB를 찾지 못했습니다. 확인 결과, 최신 MongoDB 연결 방식인 SRV 레코드(`mongodb+srv://`)를 현재 사용 중인 학교/통신사 네트워크 환경에서 'DNS 조회 단계' 자체를 방화벽으로 차단하고 있었기 때문입니다.
  * **해결:** MongoDB Atlas 대시보드에서 최신 Node.js 드라이버 대신 구버전을 선택하여, SRV를 거치지 않고 각 샤드(Shard) 서버에 직접 연결하는 **레거시 연결 문자열(`mongodb://`)**을 발급받아 네트워크 차단을 성공적으로 우회했습니다.

* **스키마 마이그레이션 중 데이터 누락 발생**
  * **원인:** 프론트엔드 화면에 D-Day 입력 달력을 만들고, DB 설계도(Schema)에 `dueDate`와 `editedAt` 공간을 추가했습니다. 그러나 백엔드 API 라우터(`app.post`, `app.put`) 코드가 업데이트되지 않아, 프론트에서 넘어온 새로운 데이터를 무시(누락)하고 제목(`title`)만 DB에 저장하고 있었습니다.
  * **해결:** 프론트엔드-API-DB로 이어지는 전체 데이터 플로우를 점검하고, 백엔드 라우터가 `dueDate`와 `editedAt` 속성을 명시적으로 받아 MongoDB에 전달하도록 파이프라인 코드를 전면 수정했습니다.

* **배포 환경 DB 쓰기 권한 차단 (MongoDB IP Whitelist)**
  * **원인:** Vercel 배포 후 웹사이트에서 데이터를 추가하려 하면 서버가 500 에러를 뱉어냈습니다. 로컬 컴퓨터의 IP만 DB 접속을 허용해 두었는데, Vercel 클라우드 서버는 접속할 때마다 유동 IP를 사용하기 때문에 MongoDB 방화벽에 막힌 것이 원인이었습니다.
  * **해결:** MongoDB Atlas의 Network Access 설정에서 IP 범위를 `0.0.0.0/0`으로 개방하여, 전 세계 어디서든 배포된 서버가 원활하게 DB에 접근할 수 있도록 권한을 수정했습니다.

<br>

#### Phase 2. 프론트엔드 환경 세팅 및 UX/UI 디테일 고도화

* **의존성 패키지 버전 충돌 (`could not determine executable to run`)**
  * **원인:** React 환경 세팅 중 Tailwind CSS 명령어(`init`)가 실행되지 않았습니다. 이는 최근 Tailwind CSS가 v4로 메이저 업데이트되면서, 튜토리얼에서 사용하던 v3의 초기화 명령어 방식이 완전히 변경/삭제되어 발생한 패키지 생태계 충돌이었습니다.
  * **해결:** 불안정한 최신 v4 패키지를 삭제하고, 명령어에 버전 태그를 명시하여 검증된 기존 버전(`npm install -D tailwindcss@3`)으로 다운그레이드 및 재설치하여 안정적인 환경을 구축했습니다.

* **반응형 웹(Responsive Web) 디자인 적용**
  * **원인:** 데스크톱 기준(고정 px)으로 화면을 개발하다 보니, 스마트폰으로 접속했을 때 입력 버튼이 패널 바깥으로 튀어나가고 UI가 심각하게 깨지는 문제가 발생했습니다.
  * **해결:** Tailwind CSS의 반응형 유틸리티 클래스(`sm:`, `md:`)를 적극 활용하여, 가로폭이 좁은 모바일 화면에서는 버튼과 입력창이 위아래(`flex-col`)로 쌓이고, 데스크톱에서는 가로(`flex-row`)로 예쁘게 배치되도록 디바이스 맞춤형 렌더링을 구현했습니다.

* **이벤트 전파(Event Bubbling) 제어 및 UX 개선**
  * **원인:** 사용자가 할 일 텍스트를 복사하기 위해 드래그하거나 빈 공간을 클릭하기만 해도 '완료' 처리가 되어버리는 치명적인 UX 불편함이 있었습니다. 클릭 이벤트(`onClick`)가 체크박스뿐만 아니라 영역 전체를 묶은 부모 태그(`div`)에 걸려 있었기 때문입니다.
  * **해결:** 부모 태그의 포괄적인 `onClick` 이벤트를 제거하고, 실제 네모난 체크박스(`input type="checkbox"`) 태그에만 상태 변화 이벤트(`onChange`)를 부여하여 타격 영역을 엄격하게 제한했습니다.

<br>

#### Phase 3. Vercel 클라우드 배포 및 CI/CD 파이프라인

* **OS 간 파일 시스템 대소문자 인식 차이 (`No such file or directory`)**
  * **원인:** Windows 기반의 내 컴퓨터에서는 폴더명을 `frontEnd`로 만들고 소문자로 호출해도 알아서 찾아갔지만, Linux 기반의 Vercel 빌드 서버는 대소문자를 칼같이 구분하기 때문에 빌드 명령어가 폴더를 찾지 못하고 뻗어버렸습니다.
  * **해결:** `vercel.json` 및 `package.json` 설정 파일 내부의 모든 경로 텍스트를 실제 폴더명인 대소문자(`frontEnd`, `backEnd`)와 100% 일치하도록 수정하여 OS 환경 차이로 인한 오류를 잡았습니다.

* **정적 파일 라우팅 매핑 오류 (지속적인 404 Not Found)**
  * **원인:** 빌드가 성공했는데도 화면에 404 에러가 떴습니다. Vite로 프론트엔드를 포장(Build)하면 결과물이 `dist` 폴더에 생성되는데, Vercel의 라우팅 설정이 이 '포장 껍데기'를 제대로 벗기지 못하고 경로를 잃어버렸기 때문입니다.
  * **해결:** Vercel의 정적 파일 제공 규칙에 맞게, `vercel.json`의 `rewrites` 목적지(destination) 경로에서 껍데기(`/dist/`) 부분을 제거하고 바로 내부 파일로 연결되도록 라우팅 매핑 구조를 직접 튜닝했습니다.

* **배포 보호(Deployment Protection) 접근 제한 해제**
  * **원인:** 배포된 URL로 접속했을 때 Vercel 로그인 창이 뜨며 일반 사용자의 접근을 막는 현상이 있었습니다. Vercel의 테스트 환경 보호 기능이 작동했기 때문입니다.
  * **해결:** 임시 주소(Preview) 대신 메인 도메인(Production)을 사용하도록 안내하고, Vercel 대시보드에서 Authentication(보안 인증) 설정을 비활성화하여 누구나 접근 가능한 퍼블릭 웹사이트로 퍼블리싱을 완료했습니다.

### 🤖 2. 생성형 AI 페어 프로그래밍 전략 (AI Utilization)
단순한 코드 생성을 넘어, 다음과 같은 방식으로 AI를 적극 활용하여 소프트웨어 공학적 시야를 넓혔습니다.

* **인프라 및 네트워크 진단 도구로 활용**
  복잡하고 긴 터미널 에러 로그를 통째로 AI에게 분석시켰습니다. 이를 통해 문제의 원인이 내 코드의 오타가 아니라 '네트워크 DNS 차단'이나 'Linux 파일 시스템의 대소문자 인식' 같은 인프라적 요인에 있다는 것을 빠르게 진단받고 올바른 디버깅 방향을 잡을 수 있었습니다.

* **풀스택 데이터 플로우 조감 및 설계**
  새로운 UI(D-Day 달력 등)를 추가할 때, 단순히 화면만 예쁘게 고치는 것을 넘어섰습니다. 프론트엔드의 상태(State), API 라우터의 수신 객체, DB 스키마 모델까지 동시에 수정해야 한다는 전체 아키텍처의 유기적인 연관성을 AI와의 문답을 통해 학습하고 구현했습니다.

* **실무적 UX/UI 고도화 지시**
  AI에게 "글자를 수정했을 때만 시간을 기록해 줘", "모바일 화면에서 버튼이 튀어나가지 않게 해 줘"와 같이 실제 서비스 기획자 관점의 구체적인 예외 조건과 반응형 설계를 지시했습니다. 이를 통해 템플릿 수준을 넘어선 실무적인 디테일을 결과물에 녹여냈습니다.

### 💡 3. 프로젝트 회고 및 엔지니어링 인사이트 (Lessons Learned)

* **프롬프트 엔지니어링의 한계와 도메인 지식의 필수성**
  AI에게 단순히 "D-Day 기능을 만들어 줘"라고 요구하는 것만으로는 데이터베이스에 값이 누락되는 치명적인 시스템 에러를 막을 수 없었습니다. 개발자가 프론트엔드-API-DB로 이어지는 **웹 아키텍처의 구조를 완벽히 이해하고 있어야만, AI가 뱉어낸 코드를 검증하고 통제하여 하나의 완성된 서비스로 조립할 수 있음**을 뼈저리게 체감했습니다.

* **로컬 환경과 프로덕션 환경의 괴리 (Local vs. Production)**
  내 컴퓨터(Local)에서 완벽하게 작동하는 코드가 클라우드 서버(Production)에 올라가는 순간, OS 파일 시스템, 네트워크 방화벽, 정적 파일 라우팅 규칙 등 인프라적 환경 차이에 의해 완전히 다른 결과를 낳을 수 있다는 배포 환경의 특수성과 무서움을 이해하게 되었습니다.

* **문제 해결을 위한 소프트웨어 공학적 시야의 확장**
  에러가 발생했을 때 단순히 코드의 오타만 찾는 1차원적인 접근에서 벗어났습니다. 통신사 DNS 정책, 패키지의 최신 메이저 업데이트 동향, 설정 파일(`vercel.json`)의 라우팅 맵핑 규칙 등 다각도로 접근하여 변수를 하나씩 통제하고 소거해 나가는 체계적인 디버깅 방법론을 체화할 수 있었습니다.

* **사용자 경험(UX)을 결정짓는 1%의 디테일**
  이벤트 전파 제어(Event Bubbling 방지), 디바이스별 반응형 디자인 렌더링, 특정 조건에서만 발동하는 시각적 피드백(완료 애니메이션) 등 사소해 보이는 프론트엔드의 디테일 처리가 실제 소프트웨어의 퀄리티와 사용자 만족도를 얼마나 극적으로 끌어올리는지 직접 경험했습니다.
